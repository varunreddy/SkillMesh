# DevOps SLI/SLO Observability Expert

Use this expert for defining Service Level Indicators, Objectives, and holistic telemetry architectures.

## When to use this expert
- You need to measure true customer experience reliability using objective data over time.
- You are facing alert fatigue and need to shift from CPU/RAM metric alerts to user-journey anomaly alerts.
- You are designing distributed tracing and distributed logging frameworks across microservices.
- You need to negotiate error budget policies between Product and Engineering.

## Execution behavior
1. Identify the critical user journeys (CUJs) driving the most business value.
2. Define Service Level Indicators (SLIs) measuring successful events vs total events (e.g. 500s vs 200s).
3. Set Service Level Objectives (SLOs) slightly below theoretical maximums (e.g. 99.9% not 100%).
4. Calculate permissible error budgets scaling continuously over rolling 28-day windows.
5. Standardize structured JSON logging topologies attaching unique trace IDs to every transaction.
6. Instrument application code with OpenTelemetry for metrics, spans, and traces.
7. Design comprehensive RED (Rate, Errors, Duration) dashboards for all major services.
8. Establish clear alerting triggers tracking the depletion burn-rate of the error budget.

## Decision tree
- If the system handles synchronous web traffic, choose latency and HTTP error rate SLIs.
- If the system is an asynchronous background queue, choose pipeline freshness, throughput, and queue depth SLIs.
- If measuring full end-to-end user workflows, choose synthetic active probing SLIs.
- If alert fatigue threatens the team, choose to alert exclusively on fast or slow error-budget burn rates.
- If investigating high latency in a specific database, choose to pivot from SLI dashboards to deep tracing spans.
- If error budgets are completely exhausted, choose to enact feature freezes until reliability is restored.

## Anti-patterns
- NEVER aim for 100% reliability SLOs (it is mathematically unachievable and prevents deployments).
- NEVER alert operators directly on transient infrastructure metrics (CPU spikes) without user impact.
- NEVER log personally identifiable information (PII) inside application telemetry traces.
- NEVER rely solely on averages for latency (always measure 95th and 99th percentiles).
- NEVER define SLOs without buy-in from both product and engineering stakeholders.
- NEVER let services communicate without forwarding correlation trace headers.

## Common mistakes
- Tracking hundreds of useless metrics leading to messy unreadable dashboards.
- Setting SLOs arbitrarily without measuring historical baseline performance first.
- Not tagging metrics with dimensionality (region, instance, tenant), making drill-downs impossible.
- Using slow aggregation queries on raw logs for real-time alerting instead of pre-computed metrics.
- Emitting unstructured multi-line text logs that defeat centralized parsing and indexing.
- Failing to adjust SLO targets downwards when external third-party dependencies clearly bound maximum uptime.

## Output contract
- Mapped Critical User Journeys (CUJs) and strict SLI definitions.
- Agreed-upon SLO thresholds and error budget calculations.
- RED method dashboard layout designs (Rate, Error, Duration).
- OpenTelemetry instrumentation specifications (Log structure, trace propagation).
- Burn-rate alerting logic thresholds isolating fast vs slow degradation.
- Error budget action policies mapping exhaustion to consequence.

## Composability hints
- Route alerts generated by burn-rate exhaustion directly into `devops.incident-response`.
- Collect infrastructure RED metrics natively from `cloud.kubernetes` and mesh sidecars.
- Work closely with `web.api-performance-caching` to isolate bottlenecks highlighted by p99 latency tracing.
- Correlate application traces with DB contention captured via queries designed in `data.sql-queries`.
